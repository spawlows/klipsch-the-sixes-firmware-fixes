# Firmware analysis

## Step 1: Configuration Storage Memory Analysis

**Hypothesis:**  
All issues are caused that the EEPROM responsible for storing device configuration may be degraded or corrupted and keep inconsistent partial random values.

**Background:**  
This is a typical issue when firmware writes configuration data too frequently, or when wear leveling is not implemented correctly (or is incorrectly implemented). Over time, this can lead to memory wear and data corruption.

**Proposed Solution:**  
- Identify the EEPROM chip used in the device  
- Replace the EEPROM with a new component  
- (Optional) Reinitialize or restore configuration data after replacement  

**Investigation Findings:**  

The device was fully disassembled and the main components were inspected. An EEPROM chip was identified during the initial analysis.

However, further investigation revealed that:
- The only EEPROM present on the board is connected to the USB audio controller.
- It is used as firmware storage for the USB audio interface.
- It has no direct physical connection to the main MCU, therefore it cannot be used for storing device configuration.

The main processor used in the device is **Microchip PIC32MX250F128D**.

This MCU does not include dedicated internal EEPROM. However, configuration data may be stored in the internal Flash memory, which is significantly more susceptible to wear compared to EEPROM.

**Endurance comparison:**
- Typical EEPROM endurance: ~1,000,000 write cycles  
- Internal Flash (PIC32MX250F128D): ~20,000 write cycles (according to documentation)

This suggests that frequent configuration writes could lead to Flash degradation over time, potentially explaining the observed issues.

## Step 2: Reverse Engineering Preparation

Prepare for reverse engineering to understand how the configuration write mechanism is implemented and whether the issue can be resolved.

Additionally, prepare for a potential MCU replacement if required after analysis.

The board was analyzed to verify that all debug pins are phisical accessible and can be connected to a debugger.

### Tools:
- MPLAB IPE v6.05 + MPLAB X IDE v6.05 with PicKit3  
- Tested also with MPLAB IPE v6.10 + MPLAB X IDE v6.10 with PicKit5  
  (both work similar and slow)
- Ghidra (reverse engineering)  
  https://github.com/NationalSecurityAgency/ghidra
- Radare2 + Cutter (used for applying binary ASM patches; GUI provided by Cutter)  
  https://cutter.re/download/
- WinMerge (compare changes)
- Notepad++
- HxD (HEX editor for inspection)
- ChatGPT (small help, mostly for corny jokes during analysis)

### Knowledge on begin:
- Embedded C programming knowledge (not familiar with Microchip PIC)
- Basic understanding of assembly concepts (stack, jumps, functions, return, etc.)  
  (experience with 8051 and debugging compiled code on x86/ARM)
- No prior experience with Microchip PIC
